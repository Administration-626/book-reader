# AI 逐页读书技能

## 目标

把一本书从“文件”变成“可读的知识结构”：

1. 识别格式并提取正文
2. 按页/章分批理解
3. 累积主题、概念、论证和例子
4. 生成结构化读书解释
5. 保存为本地 Markdown

## 触发时机

只要用户要做下面任一件事，就用这个技能：

- 读一本书并做总结、导读、解释、笔记、知识提取
- 分析 PDF/EPUB/MOBI/TXT 书稿
- 解释章节、主题、观点、论证结构
- 生成本地 Markdown 读书笔记

## 输入判断

先确定三件事：

1. 输入来源：本地文件、URL、还是用户上传
2. 文件格式：PDF、EPUB、MOBI、TXT
3. 目标粒度：快速解释、章节级总结、还是完整笔记

同时确定书籍类型：

- 现代论著：按章节论证、概念和例子阅读
- 问题导向型导论书：按“问题链 -> 概念工具 -> 章节答案”阅读
- 古籍/经典：区分原文、注释、译文、评述和版本背景
- 注译本：不要把注释、译文、前言、参考书目混成作者原意

如果用户没有明确要求输出格式，默认生成：

- 先给简短口头解释
- 再保存一份本地 Markdown 读书笔记

## 标准流程

### 1) 获取文件

- 本地已有文件：直接使用
- URL：下载到 `workspace/`
- 上传文件：放到 `workspace/`

确认文件可读后再继续。

### 2) 提取文本

优先使用仓库里的提取脚本：

```bash
python3 scripts/extract_book.py <book_path> -o workspace/<book_slug>_extracted.json
```

输出文件名用简短书名，避免长文件名和覆盖旧的 `book_extracted.json`。

如果缺依赖，先尝试脚本内置 fallback；fallback 失败后再按格式安装最小依赖：

```bash
python3 -m pip install pymupdf pypdf ebooklib mobi beautifulsoup4
```

### 3) 验证提取结果

重点检查：

- `total_pages` / `total_chapters`
- `processed_pages`
- `content_pages`
- `skipped_pages`

还要抽查前 3-5 个内容单元：

- 标题是否像真实章节
- 前言、目录、参考书目、后记是否被误纳入主线
- EPUB HTML 顺序是否符合目录或正文顺序
- 注译本是否能看出原文/注释/翻译的边界

如果 EPUB 提取出的标题大量重复为书名或“未知”：

- 回查 `toc.ncx` / `content.opf` / 正文前几行
- 用目录标题或正文首个有效小标题重建章节映射
- 在笔记里记录“标题已人工校准”的阅读口径
- 不要直接按错误标题生成章节解释

如果 `content_pages == 0` 或极少：

- 可能是扫描版 PDF
- 可能文件损坏
- 可能是目录/版权页为主

这时先告诉用户结果受限，不要硬编内容。

### 4) 分批理解

不要一次性把整本书塞进上下文。按规模分批处理：

- 短书：每 1-3 单元一批
- 中书：每 5-8 单元一批
- 长书：每 8-12 单元一批

每批只做四件事：

- 判断这批是核心内容还是辅助内容
- 提炼核心观点
- 记录关键概念/术语/例子
- 更新全书累计摘要

古籍/经典类每批额外记录：

- 这批属于原文、注释、译文、序跋、书信还是后人评述
- 核心术语在本批中的用法变化
- 是否存在容易被现代读者误读的地方

问题导向型导论书每批额外记录：

- 本批提出的问题是什么
- 本批使用哪些概念工具
- 本批如何改变前面的问题链
- 哪些内容是练习题、附录或教学辅助，不应冒充正文主线

上下文里只保留最近 10-15 个核心要点，避免膨胀。

### 5) 生成解释

解释时优先回答这些问题：

- 这本书在问什么
- 作者的答案结构是什么
- 每章和全书主线如何连接
- 读者最容易误解哪里
- 哪些概念最关键

解释古籍时优先给出：

- 术语关系，而不是只翻译字面意思
- 思想主线，而不是逐条堆砌摘录
- 实践入口，而不是只做学术背景介绍

解释问题导向型导论书时优先给出：

- 全书问题链，而不是只列章节摘要
- 每章如何推进、改写或挑战前一章的问题
- 读者可带走的思考方法和复读路线

## 输出标准

最终输出应至少包含：

- 书籍概览
- 一句话总结
- 全书主线
- 章节解释
- 关键概念表
- 全书总结
- 可继续追问的问题

古籍/经典类读书笔记至少补充：

- 阅读口径：哪些内容纳入主线，哪些只作版本背景
- 核心论证链：从问题到答案的递进关系
- 误读提醒：区分作者主张、译注者解释和读者推断

重生成已有笔记时：

- 先确认会覆盖哪个 Markdown 文件
- 新解析 JSON 使用书名专属文件名
- 在新笔记中说明相对旧版本的结构校准点

如果用户只要“解释”，就优先给简洁自然语言。
如果用户要“读书笔记”，就同时生成 Markdown 文件。

## 写作规则

- 默认使用书籍语言；不明确时优先使用用户语言
- 不要长篇照抄原文
- 只保留短引用，且只在确实必要时使用
- 把“事实总结”和“个人解释/推断”分开写
- 用章节标题、概念表和要点列表组织输出

## 常见故障

- **扫描版 PDF**：提取不到正文，提示需要 OCR
- **EPUB 章节顺序混乱**：以提取顺序为准，并说明可能有偏差
- **MOBI 失败**：建议先转 EPUB
- **依赖缺失**：只装当前格式需要的最小依赖
- **文件名过长或含非法字符**：保存为 `workspace/book_notes.md`

## 进度汇报

处理时主动给用户短进度：

- 正在解析文件
- 已提取多少内容单元
- 正在分批总结
- 正在生成最终笔记
- 已保存到哪个路径

## 质量检查

完成前检查这五项：

1. 文件格式和页数/章节数正确
2. 提取内容不是空的
3. 章节主线说得通
4. 没有长篇引用原文
5. 输出文件已经落到本地路径
